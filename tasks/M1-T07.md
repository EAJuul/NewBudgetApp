## Public API to produce
```dart
// lib/data/database/tables/category_budgets_table.dart
@DataClassName('CategoryBudgetRow')
class CategoryBudgets extends Table {
  TextColumn get id => text()();
  TextColumn get categoryId => text()();
  TextColumn get month => text()();            // 'YYYY-MM'
  IntColumn get assigned => integer()();        // milliunits; may be negative
  TextColumn get createdAt => text()();
  TextColumn get updatedAt => text()();

  @override
  Set<Column> get primaryKey => {id};

  @override
  List<Set<Column>> get uniqueKeys => [
        {categoryId, month},
      ];
}

// lib/data/database/tables/targets_table.dart
class Targets extends Table {
  TextColumn get id => text()();
  TextColumn get categoryId => text()();
  TextColumn get type => textEnum<TargetType>()();
  IntColumn get amount => integer()();
  TextColumn get targetMonth => text().nullable()();   // 'YYYY-MM'
  TextColumn get createdAt => text()();
  TextColumn get updatedAt => text()();

  @override
  Set<Column> get primaryKey => {id};

  @override
  List<Set<Column>> get uniqueKeys => [
        {categoryId},
      ];
}
```
Annotate `Targets` with `@DataClassName('TargetRow')` (same convention as
`CategoryBudgets` above).

## Implementation notes
- **`@DataClassName` is required on both classes.** It makes Drift name the
  generated row classes `CategoryBudgetRow` / `TargetRow`. Without it Drift
  generates `CategoryBudget` / `Target`, which collide with the domain entities
  of the same name (M1-T18, M1-T23). Every table in this schema uses the
  `<Entity>Row` convention.
- `uniqueKeys` enforces `UNIQUE(category_id, month)` on `category_budgets` and
  `UNIQUE(category_id)` on `targets` (0..1 target per category — `docs/03`).
- `assigned` may be **negative** — the user can pull money back to RTA
  (`docs/04` invariant #4). Add no non-negative constraint.
- `targetMonth` is nullable — only `TargetType.targetBalanceByDate` uses it.
- `categoryId` on both tables is a logical FK to `categories.id`. Define it as
  plain `text()()` **without** `.references()` — the referenced `Categories`
  table is owned by a different card and the wiring task M1-T11 adds every
  cross-table FK constraint centrally. **NEEDS REVIEW:** M1-T11.
- Timestamps are ISO-8601 text; use `TextColumn`, never `DateTimeColumn`
  (`docs/03` — Dates).
- Do **not** add these tables to `AppDatabase.tables` — that is M1-T11.

## Tests to write
File: `test/data/database/tables/category_budgets_targets_test.dart`

Define a test-only database in the test file:
```dart
@DriftDatabase(tables: [CategoryBudgets, Targets])
class _TestDb extends _$_TestDb {
  _TestDb() : super(NativeDatabase.memory());
  @override
  int get schemaVersion => 1;
}
```

Test cases:
- Insert a `CategoryBudgetsCompanion` with a positive `assigned`; select it back
  and verify the value.
- Insert a `category_budgets` row with a **negative** `assigned`; verify it
  round-trips.
- Insert two `category_budgets` rows with the same `categoryId` **and** `month`;
  the second insert throws (UNIQUE violation).
- Insert a `TargetsCompanion` for `TargetType.monthlyFunding` with `targetMonth`
  null and one for `TargetType.targetBalanceByDate` with a non-null
  `targetMonth`; verify `type` and `targetMonth` round-trip.
- Insert two targets with the same `categoryId`; the second insert throws.
