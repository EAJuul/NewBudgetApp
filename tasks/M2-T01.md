## Goal

(accounts, categories, transactions, …) in an in-memory database. Every
M2 engine test asserts exact milliunit results against data built with it.
This is the project's single most important test asset (`docs/07`).

## Public API to produce
```dart
// test/support/budget_fixture.dart
class BudgetFixture {
  /// Spins up an in-memory database wired through the M1-T26 providers and
  /// seeds one `Budget` row. `budgetId` defaults to a fixed test value.
  static Future<BudgetFixture> create({String budgetId});

  String get budgetId;

  // Repositories — for service-level tests (M2-T07+).
  AccountRepository get accounts;
  CategoryRepository get categories;
  CategoryBudgetRepository get categoryBudgets;
  TransactionRepository get transactions;
  TargetRepository get targets;
  ScheduledTransactionRepository get schedules;

  // Builders — persist a row, return the domain entity. Ids auto-generate
  // (uuid) when not supplied.
  Future<Account> addAccount({
    String? id,
    String name,
    AccountType type,
    bool closed,
  });
  Future<CategoryGroup> addGroup({String? id, String name, SystemGroupType? systemType});
  Future<Category> addCategory({
    String? id,
    required String groupId,
    String name,
    String? linkedAccountId,
  });
  Future<CategoryBudget> assign({
    required String categoryId,
    required MonthKey month,
    required Money amount,
  });
  Future<Target> addTarget({
    required String categoryId,
    required TargetType type,
    required Money amount,
    MonthKey? targetMonth,
  });
  Future<Transaction> addTransaction({
    String? id,
    required String accountId,
    required DateTime date,
    required Money amount,
    String? categoryId,
    ClearedStatus cleared,
    bool isSplit,
    List<SubTransaction> subTransactions,
  });

  // Snapshots — for the pure calculator tests (M2-T02..T05).
  Future<List<Account>> allAccounts();
  Future<List<Transaction>> allTransactions();
  Future<List<SubTransaction>> subTransactionsOf(String transactionId);

  /// Closes the database and disposes the provider container.
  Future<void> dispose();
}
```

## Implementation notes
- `create` builds `AppDatabase.forTesting(NativeDatabase.memory())`, then a
  `ProviderContainer` overriding `appDatabaseProvider` with that database, reads
  the nine repositories from the container, and seeds one `Budget` (so account
  inserts satisfy the `budgets` foreign key).
- Builders write through the **repository interfaces**, not raw Drift — this
  exercises the real mappers and keeps the fixture honest.
- `addAccount` defaults `type` to `AccountType.checking`; `onBudget` follows
  `type.isOnBudget`.
- `addTransaction` with `isSplit: true` saves the parent and the supplied
  `subTransactions` together (via `TransactionRepository.save`).
- Auto-generate UUIDs with `package:uuid` when an `id` argument is omitted.
- `dispose` must be registered by callers with `addTearDown(fixture.dispose)`.
- The fixture lives under `test/`; engine tests import it by relative path
  (e.g. `import '../../support/budget_fixture.dart';`).

## Tests to write
File: `test/support/budget_fixture_test.dart`
- `create` returns a fixture whose `allAccounts` / `allTransactions` are empty.
- After `addAccount` + `addCategory` + `addTransaction`, the snapshots return
  the built rows with the expected `Money` amounts and `MonthKey` values.
- A split built via `addTransaction(isSplit: true, subTransactions: [...])` is
  retrievable via `subTransactionsOf`.
- `dispose` closes the database without error.
