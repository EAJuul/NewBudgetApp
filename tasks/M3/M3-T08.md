## Goal
Let the user permanently delete an account, but only when it is safe: an
account that still has transactions must not be deletable, to avoid orphaned
history. The user is told to clear or move the transactions first.

## Public API to produce
```dart
// added to account_list_controller.dart
/// Outcome of a delete attempt.
enum AccountDeletionResult { deleted, blockedHasTransactions }

// added to class AccountListController
/// Deletes the account only when it has no transactions. Returns
/// [AccountDeletionResult.blockedHasTransactions] without deleting otherwise.
Future<AccountDeletionResult> deleteAccount(String accountId);
```

## Implementation notes
- `deleteAccount` first checks for transactions:
  `final txns = await transactionRepository.watchByAccount(accountId).first;`
  If `txns.isNotEmpty`, return `AccountDeletionResult.blockedHasTransactions`
  and do **not** delete.
- Otherwise call `accountRepository.delete(accountId)` and return
  `AccountDeletionResult.deleted`.
- The guard is a controller-level business rule — it returns a result enum
  rather than throwing. Do **not** introduce a typed exception or a
  `core/error/` type for this; a missing-account case is not expected here
  because the control is reached from the loaded edit screen.
- The watched list refreshes automatically once the row is gone — no manual
  `state` update.
- **UI control** in `edit_account_screen.dart`: add a destructive-styled
  "Delete account" button beneath the close / reopen control (M3-T07).
  - On press, show a confirming `AlertDialog`.
  - On confirm, call `deleteAccount`. If the result is `deleted`, navigate
    back to `/accounts` and show a `SnackBar`. If it is
    `blockedHasTransactions`, show an explanatory `SnackBar`/dialog
    ("This account still has transactions — move or delete them first.") and
    stay on the screen.

## Tests to write
File: `test/features/accounts/presentation/account_delete_test.dart`

`ProviderContainer` with `appDatabaseProvider` overridden to an in-memory
database; seed a `Budget` and accounts/transactions through the repositories.
- `deleteAccount` on an account with no transactions returns `deleted` and the
  account is gone from the controller's list afterward.
- `deleteAccount` on an account that has a transaction returns
  `blockedHasTransactions` and the account still exists.
- After the transaction is removed, a second `deleteAccount` call succeeds.
