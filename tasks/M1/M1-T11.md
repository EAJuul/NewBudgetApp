## Public API to produce
```dart
// lib/data/database/app_database.dart  (edited)
@DriftDatabase(
  tables: [
    Budgets,
    Accounts,
    CategoryGroups,
    Categories,
    CategoryBudgets,
    Targets,
    Payees,
    Settings,
    Transactions,
    SubTransactions,
    ScheduledTransactions,
  ],
)
class AppDatabase extends _$AppDatabase {
  AppDatabase() : super(openConnection());
  AppDatabase.forTesting(super.e);

  @override
  int get schemaVersion => 1;

  @override
  MigrationStrategy get migration => MigrationStrategy(
        onCreate: (m) => m.createAll(),
        beforeOpen: (details) async {
          await customStatement('PRAGMA foreign_keys = ON');
        },
      );
}
```
The `@riverpod AppDatabase appDatabase(Ref ref)` provider already in the file is
unchanged.

## Implementation notes
- Add a `part`-less `import` for each table file at the top of
  `app_database.dart`; the `@DriftDatabase` annotation references the eleven
  table classes by name.
- **Deferred foreign keys.** Each table card left cross-table FK columns as
  plain `text()` columns and tagged them `NEEDS REVIEW: M1-T11`. Convert each to
  `.references(<Table>, #id)` — e.g. in `transactions_table.dart`,
  `TextColumn get accountId => text().references(Accounts, #id)();`. The Edit
  list above names every column. Import the referenced table file where needed.
- `transferTransactionId` / `sub_transactions.transactionId` already carry
  `.references` from M1-T09 — do not duplicate them.
- `PRAGMA foreign_keys = ON` in `beforeOpen` makes SQLite enforce the FKs;
  without it Drift declares but does not enforce them.
- After editing, run `dart run build_runner build --delete-conflicting-outputs`
  so `app_database.g.dart` regenerates with all eleven tables.
- **Schema export.** Create the `drift_schema/` directory, then run
  `dart run drift_dev schema dump lib/data/database/app_database.dart drift_schema/`.
  Commit the resulting JSON file (drift writes a versioned name such as
  `drift_schema_v1.json`). This is the baseline future migration tests upgrade
  from (`docs/07`).
- `schemaVersion` stays `1`. Any future schema change increments it and adds an
  `onUpgrade` step here.

## Tests to write
File: `test/data/database/app_database_schema_test.dart`

Use the real database in-memory: `AppDatabase.forTesting(NativeDatabase.memory())`
in `setUp`, `db.close()` in `tearDown`.

Test cases:
- The database opens and `schemaVersion` is `1`.
- A `SELECT` against every one of the eleven tables succeeds and returns an
  empty result (proves each table was created by `createAll`).
- Insert a `budget`, then an `account` referencing that budget — succeeds.
- Insert an `account` whose `budgetId` does not exist — the insert throws
  (proves `PRAGMA foreign_keys = ON` and the FK are active).
- Insert a `transaction` whose `accountId` does not exist — the insert throws.
