## Goal
Wire dependency injection: one `@riverpod` provider per repository / store,
each returning the **domain interface** type and constructed from
`appDatabaseProvider`. This is the seam the presentation layer reads from.

## Implementation notes
- Each provider's **return type is the domain interface** (`AccountRepository`,
  not `AccountRepositoryImpl`). Consumers depend only on the interface; the
  concrete class is an implementation detail of this file.
- Every impl takes one `AppDatabase` constructor argument (M1-T14..M1-T25), so
  every provider body is the same one-liner over `appDatabaseProvider`.
- This file is the project's **DI seam**. It lives in `lib/data/` — the same
  layer as `appDatabaseProvider` (M1-T04) — and is the one `data/` file the
  presentation layer is allowed to import. Controllers `ref.watch` these
  providers; they never import a `*_impl.dart` directly.
- Use the `Ref` type from `riverpod_annotation` (matching `appDatabase` in
  M1-T04). Run `build_runner` to generate `repository_providers.g.dart`.
- Do not add caching, `keepAlive`, or disposal logic — the impls are cheap and
  stateless; `appDatabaseProvider` already owns the database lifecycle.

## Tests to write
File: `test/data/repository_providers_test.dart`

Build a `ProviderContainer` with `appDatabaseProvider` overridden to an
in-memory database:
```dart
final container = ProviderContainer(
  overrides: [
    appDatabaseProvider.overrideWith(
      (ref) => AppDatabase.forTesting(NativeDatabase.memory()),
    ),
  ],
);
addTearDown(container.dispose);
```

Test cases:
- Reading each of the nine providers returns a non-null instance.
- Each provider's value `is` the expected domain interface type (e.g.
  `container.read(accountRepositoryProvider) is AccountRepository`).
- Reading the same provider twice returns the identical instance (Riverpod
  caches it within the container).
