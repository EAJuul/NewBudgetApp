## Public API to produce
```dart
// lib/data/daos/accounts_dao.dart
import 'package:budget_app/data/database/app_database.dart';
import 'package:budget_app/data/database/tables/accounts_table.dart';
import 'package:drift/drift.dart';

part 'accounts_dao.g.dart';

@DriftAccessor(tables: [Accounts])
class AccountsDao extends DatabaseAccessor<AppDatabase> with _$AccountsDaoMixin {
  AccountsDao(super.db);

  /// All accounts in [budgetId], ordered by `sortOrder`, re-emitted on change.
  Stream<List<AccountRow>> watchByBudget(String budgetId);

  Future<AccountRow?> findById(String id);

  /// Insert or update by primary key.
  Future<void> upsert(AccountsCompanion account);

  Future<void> deleteById(String id);
}
```

## Implementation notes
- `AccountsDao` is a standalone `@DriftAccessor` constructed as
  `AccountsDao(db)`; it is **not** registered in `@DriftDatabase(daos: [...])`.
  Run `build_runner` to generate `_$AccountsDaoMixin` in `accounts_dao.g.dart`.
- `watchByBudget` uses `select(accounts)..where(...)..orderBy(...)` then
  `.watch()` so callers get a reactive `Stream`.
- `upsert` uses `into(accounts).insertOnConflictUpdate(account)`.
- The generated row class is `AccountRow` and the companion is
  `AccountsCompanion` (see M1-T05 — `@DataClassName`).
- The DAO returns Drift row types (`AccountRow`); converting them to the domain
  `Account` entity is the repository's job (M1-T14), not the DAO's.
- This DAO does no timestamp or UUID logic — callers supply complete companions.

## Tests to write
File: `test/data/daos/accounts_dao_test.dart`

Use a real in-memory database (`docs/07` — never mock the database):
```dart
late AppDatabase db;
late AccountsDao dao;
setUp(() {
  db = AppDatabase.forTesting(NativeDatabase.memory());
  dao = AccountsDao(db);
});
tearDown(() => db.close());
```
Because `PRAGMA foreign_keys = ON` is active (M1-T11), insert a parent `budgets`
row before inserting accounts.

Test cases:
- `upsert` a new account, then `findById` returns a row with matching fields.
- `upsert` an account with an existing `id` but a changed `name`; `findById`
  shows the updated name and there is still only one row.
- `watchByBudget` emits the accounts for that budget, ordered by `sortOrder`,
  and excludes accounts of another budget.
- `deleteById` removes the row; `findById` then returns `null`.
- `findById` for an unknown id returns `null`.
