## Goal
The controller behind both the create-account screen (M3-T05) and the
edit-account screen (M3-T06). It persists a new account — together with a
starting-balance transaction — and updates an existing account's editable
fields. A "form controller" covers the whole add/edit form, so one controller
serves both screens.

## Public API to produce
```dart
// lib/features/accounts/presentation/account_form_controller.dart
part 'account_form_controller.g.dart';

/// Drives the add/edit account form. `state` is `AsyncValue<void>`: `data`
/// when idle or after a successful write, `loading` mid-write, `error` on
/// failure — screens render it for progress and error feedback.
@riverpod
class AccountFormController extends _$AccountFormController {
  @override
  FutureOr<void> build() {}

  /// Creates a new account in the primary budget. When [startingBalance] is
  /// non-zero, also writes a starting-balance transaction on that account.
  /// Returns the new account's id.
  Future<String> create({
    required String name,
    required AccountType type,
    required Money startingBalance,
  });

  /// Updates an existing account's editable fields (name, type, note).
  /// The starting balance is not editable here — it lives in a transaction.
  Future<void> update({
    required String accountId,
    required String name,
    required AccountType type,
    String? note,
  });
}
```

## Implementation notes
- Callers pass already-validated input (the screens validate the form). Assume
  `name` is non-empty and trimmed.
- **`create`:**
  1. resolve the primary budget — `budgetRepository.getPrimary()`; if `null`,
     fail with a `StateError` ("no budget").
  2. `sortOrder` for the new account = the current account count for that
     budget (`accountRepository.watchAll(budgetId).first.length`).
  3. build `Account(id: <uuid>, budgetId: budget.id, name: name, type: type,
     onBudget: type.isOnBudget, closed: false, note: null, sortOrder: …)` and
     `accountRepository.save(account)`.
  4. if `!startingBalance.isZero`, build a `Transaction(id: <uuid>,
     accountId: account.id, date: DateTime.now(), amount: startingBalance,
     payeeId: null, categoryId: null, memo: 'Starting Balance',
     cleared: ClearedStatus.cleared, approved: true, flagColor: null,
     transferTransactionId: null, transferAccountId: null,
     scheduledTransactionId: null, importId: null, isSplit: false,
     deleted: false)` and `transactionRepository.save(transaction)`.
  5. return `account.id`.
- A `categoryId == null` transaction on an on-budget account is exactly what
  the RTA calculator (M2-T05) counts as inflow-to-RTA — no special category id
  is needed. Off-budget accounts: the same null-category transaction is simply
  not counted toward RTA. Do not branch on `type`.
- **`update`:** `accountRepository.findById(accountId)`; if `null`, fail with a
  `StateError`. `copyWith(name: name, type: type, onBudget: type.isOnBudget,
  note: note)`; `accountRepository.save(updated)`. Preserve `closed` and
  `sortOrder` unchanged.
- Wrap each method body so `state` reflects progress:
  `state = const AsyncLoading();` then
  `state = await AsyncValue.guard(() async { … });` — and still return the id /
  void to the caller. If `state` is an error afterward, rethrow so the caller
  sees it too, or have the screen read `state`. Keep one consistent approach
  and document it in a one-line comment.
- Generate UUIDs with `package:uuid`.
- Run `build_runner` for `account_form_controller.g.dart`.

## Prerequisite note
This controller needs no money **formatting** — it receives `Money` directly.
Parsing the user's balance text into `Money` is the create screen's job
(M3-T05). This card is self-contained.

## Tests to write
File: `test/features/accounts/presentation/account_form_controller_test.dart`

`ProviderContainer` with `appDatabaseProvider` overridden to an in-memory
database; seed a `Budget` first.
- `create` with `Money.zero()` starting balance: one account is persisted, no
  transaction exists on it.
- `create` with a non-zero starting balance: the account exists and a single
  `Transaction` of exactly that `Money` amount, `categoryId == null`,
  exists on the account.
- `create` returns the id of the account it persisted.
- Two `create` calls produce `sortOrder` 0 then 1.
- `update` changes `name` / `type` / `note` and leaves `closed` and
  `sortOrder` untouched; `onBudget` follows the new `type`.
- `update` on an unknown id surfaces an error in `state`.
- `create` with no `Budget` seeded surfaces an error in `state`.
