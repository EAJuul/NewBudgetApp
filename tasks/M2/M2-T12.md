## Goal
Extend the category-available calculator with the **full** rollover rule: cash
overspending is absorbed by Ready-to-Assign (carryover clamps to zero); credit
overspending rolls forward in the category. The simple mode (M2-T04) stays
available.

## Public API to produce

```dart
// lib/domain/budgeting/category_available_calculator.dart  (edited)
Map<MonthKey, Money> computeCategoryAvailableSeries({
  required List<MonthKey> months,
  required Money Function(MonthKey month) assignedFor,
  required Money Function(MonthKey month) activityFor,
  // NEW — optional. When null, the simple rollover of M2-T04 is used unchanged.
  // When supplied, returns the credit-overspent portion of [month] (>= 0).
  Money Function(MonthKey month)? creditOverspentFor,
});
```

## Implementation notes
- The running computation `available(m) = carryover(m-1) + assigned(m) +
  activity(m)` is unchanged. Only **`carryover`** — what passes from month `m`
  to month `m+1` — changes:
  - `available(m) >= 0` → `carryover = available(m)` (as before).
  - `available(m) < 0` and `creditOverspentFor == null` → `carryover =
    available(m)` (simple mode — all negatives roll forward, M2-T04 behaviour).
  - `available(m) < 0` and `creditOverspentFor != null` → the negative is split:
    the **credit** part rolls forward, the **cash** part is dropped (absorbed by
    RTA in month `m`). `carryover = −min(creditOverspentFor(m),
    available(m).abs())`. Equivalently `carryover = max(available(m),
    −creditOverspentFor(m))`.
- `creditOverspentFor(m)` is a non-negative `Money` — the sum of
  `creditOverspendingPortion` (M2-T13) across the category's credit-card
  purchases dated in month `m`. The caller (M2-T07 / a later wiring task)
  supplies it; this function only consumes the closure.
- The per-month `available` value stored in the returned map is the *raw*
  `carryover(m-1) + assigned + activity` (it may be negative). The cash/credit
  rule changes what the **next** month inherits, not the displayed value of the
  overspent month itself.
- Keep both modes in one function via the nullable parameter — do not fork a
  second function. `Money` arithmetic only.

## Tests to write
File: `test/domain/budgeting/category_available_overspending_test.dart`
Hand-computed milliunit values; both modes.
- **Simple mode** (no `creditOverspentFor`): a negative month still rolls its
  full negative forward — re-confirms M2-T04 behaviour.
- **Cash overspending**: `available(m) < 0`, `creditOverspentFor(m) == zero` →
  next month's carryover is zero (the cash overspend does not roll forward).
- **Credit overspending**: `available(m) < 0`, `creditOverspentFor(m)` equals
  the whole overspend → the full negative rolls forward.
- **Mixed**: overspend is part cash, part credit → only the credit part rolls
  forward; the cash part is dropped.
- A positive month is unaffected by `creditOverspentFor`.
